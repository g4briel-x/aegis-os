# Execution Contract Safety Model

Version: v0.6.0  
Status: safety foundation  
Branch: develop/v0.6-runtime

## Role of this file

This file defines the safety model used by Aegis OS execution contracts.

The safety model determines whether a contract may be planned, simulated, submitted for approval or blocked.

## Purpose

Aegis OS must evaluate safety before allowing executable behavior.

Every execution contract declares a safety level that communicates the maximum level of action currently permitted.

## Safety levels

The runtime currently defines four safety levels:

```text
safe-read-only
safe-dry-run
requires-approval
blocked
```

## safe-read-only

Role: allow operations that inspect information without modifying state.

Examples:

```text
read registry files
inspect asset metadata
list related assets
generate reports from existing data
validate structures
```

A read-only operation must not:

```text
modify files
delete files
call external systems with side effects
execute arbitrary shell commands
```

## safe-dry-run

Role: allow planning and simulation without executing real actions.

This is the default safety level generated by the v0.6 contract builder.

Allowed behavior:

```text
create an execution plan
resolve dependencies
validate readiness
produce a dry-run report
describe intended actions
```

Forbidden behavior:

```text
write or modify files
delete resources
send messages
call external APIs
execute shell commands
run untrusted code
```

## requires-approval

Role: declare that an operation cannot proceed without explicit authorization.

Examples of future approval-sensitive actions:

```text
modifying repository files
creating external resources
sending email
calling an AI provider
calling third-party APIs
publishing artifacts
executing deployment actions
```

In v0.6, this level does not authorize execution. It only signals that approval would be required in a future runtime.

## blocked

Role: explicitly prevent execution.

A blocked contract must fail validation.

Typical reasons:

```text
unsafe action request
unsupported execution mode
untrusted code
missing governance approval
policy violation
restricted asset
```

## Forbidden action catalog

The current validator recognizes these restricted actions:

```text
delete_file
delete_directory
write_file
modify_file
send_email
call_external_api
execute_shell_command
run_untrusted_code
```

The contract builder adds these actions to generated contracts as explicit prohibitions.

## Safe default policy

Aegis OS follows a deny-by-default execution policy.

When a contract is generated automatically:

```text
safety level: safe-dry-run
allowed modes: plan, dry-run
real execution: disabled
destructive operations: forbidden
external actions: forbidden
```

## Validation behavior

The validator produces an error when:

```text
the contract safety level is blocked
an unsupported execution mode is declared
the asset identifier is missing
```

The validator produces a warning when:

```text
dry-run is not explicitly allowed
no forbidden actions are declared
an unknown forbidden action is declared
```

## Current safety boundary

Aegis OS v0.6 does not execute real actions.

The current runtime is limited to:

```text
planning
inspection
contract generation
contract validation
dry-run simulation
```

## Future safety controls

Future versions may add:

```text
approval tokens
role-based authorization
policy-as-code checks
tool allowlists
provider allowlists
resource limits
execution timeouts
sandboxing
audit logs
rollback requirements
human confirmation
```

## Final principle

> No real action should occur unless the execution contract, safety policy and approval requirements explicitly permit it.
