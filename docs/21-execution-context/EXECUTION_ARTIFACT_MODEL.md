# Execution Artifact Model

Version: v0.6.0  
Status: artifact declaration foundation  
Branch: develop/v0.6-runtime

## Role of this file

This file documents the Aegis OS execution artifact model.

Execution artifacts represent outputs expected from an execution contract and attached to an execution context before any real action occurs.

## Runtime implementation

The artifact model is declared in:

```text
runtime/src/aegis_runtime/execution/context.py
```

Artifact declarations are generated by:

```text
runtime/src/aegis_runtime/execution/context_builder.py
```

Contract output definitions originate from:

```text
runtime/src/aegis_runtime/execution/contracts.py
```

## Purpose

The artifact model connects contract outputs to the execution context.

It answers these questions:

```text
Which outputs are expected?
What type of artifact is declared?
Where may the artifact be stored?
Which metadata describes it?
Has the artifact been declared or produced?
```

## ExecutionArtifact structure

An `ExecutionArtifact` contains:

```text
name
artifact_type
location
metadata
```

### name

The `name` identifies the artifact.

Examples:

```text
execution_plan
dry_run_report
```

The name is inherited from an `ExecutionOutput` declared by the execution contract.

### artifact_type

The `artifact_type` describes the artifact category.

Generated artifacts currently use:

```text
declared-output
```

This means the output is expected but has not yet been persisted or produced.

### location

The `location` field may contain a future file path, URI or storage identifier.

Current default:

```text
empty string
```

A blank location confirms that no artifact has been written to disk.

### metadata

The metadata mapping contains additional artifact information.

Generated artifacts currently include:

```text
description
status
```

Default status:

```text
declared
```

## Artifact generation flow

The current flow is:

```text
1. Resolve the target asset.
2. Build the execution contract.
3. Read the contract outputs.
4. Create one ExecutionArtifact per output.
5. Mark each artifact as declared.
6. Attach the artifacts to the ExecutionContext.
```

## Example contract outputs

```text
execution_plan
dry_run_report
```

## Example generated artifacts

```text
name: execution_plan
artifact_type: declared-output
location: ""
metadata:
  description: Structured execution plan generated for the asset.
  status: declared
```

```text
name: dry_run_report
artifact_type: declared-output
location: ""
metadata:
  description: Safe dry-run report showing what would be executed.
  status: declared
```

## Artifact lifecycle

The current v0.6 lifecycle supports only:

```text
declared
```

Possible future lifecycle states may include:

```text
declared
prepared
generated
validated
persisted
published
failed
discarded
```

These future states are not yet implemented.

## Context integration

Artifacts are stored in:

```python
ExecutionContext.artifacts
```

An artifact may also be attached manually through:

```python
context.add_artifact(artifact)
```

The context serializes artifacts through:

```python
context.to_dict()
```

## CLI visibility

Declared artifacts are displayed by:

```powershell
python -m aegis_runtime execution context <asset-id>
```

Example:

```text
Declared artifacts:
- execution_plan [type=declared-output]
- dry_run_report [type=declared-output]
```

JSON output:

```powershell
python -m aegis_runtime --json execution context <asset-id>
```

## Automated tests

Artifact declarations are tested in:

```text
tests/runtime/test_execution_context.py
```

The tests verify:

```text
one artifact is generated per contract output
artifact names match contract output names
context serialization includes artifacts
artifact count is correct
```

## Current safety boundary

Execution artifacts are declarations only.

The current implementation does not:

```text
create files
write reports
modify repository content
upload data
call external storage
publish artifacts
delete artifacts
```

Artifact construction is therefore safe for planning and dry-run contexts.

## Current limitations

The v0.6 artifact model does not yet provide:

```text
artifact identifiers
content payloads
checksums
MIME types
file sizes
creation timestamps
producers
consumers
validation status
persistence policies
retention policies
access control
```

## Future evolution

Future versions may introduce:

```text
artifact manifests
artifact persistence
content validation
checksums
storage providers
workspace-relative paths
temporary artifacts
sensitive artifact classification
artifact approval requirements
artifact audit events
```

## Final principle

> An artifact must remain a declaration until the runtime explicitly authorizes, generates, validates and records it.
