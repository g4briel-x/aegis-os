# Pattern Quality Gate

Version: 0.1.0  
Status: Foundation Draft

---

# 1. Purpose

This document defines the minimum quality criteria for accepting a Pattern into Aegis OS.

---

# 2. Required Checks

A Pattern passes the quality gate only if:

```text
[ ] Problem is clearly stated
[ ] Context is defined
[ ] Recommended solution is specific
[ ] Trade-offs are documented
[ ] Risks are documented
[ ] Examples are included
[ ] Validation checklist exists
[ ] Metadata is complete
[ ] Related skills or playbooks are linked
[ ] Naming follows repository conventions
```

---

# 3. Review Criteria

Reviewers should assess:

- usefulness;
- reusability;
- clarity;
- correctness;
- completeness;
- domain alignment;
- maintainability.

---

# 4. Rejection Reasons

Reject a Pattern when:

- it is too vague;
- it duplicates an existing Pattern;
- it lacks context boundaries;
- it has no trade-offs;
- it is really a Playbook;
- it is not reusable.

---

# 5. Final Principle

> A Pattern should be reusable enough to deserve a place in the system.
