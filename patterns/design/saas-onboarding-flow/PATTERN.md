# SaaS Onboarding Flow Pattern

Version: 0.1.0  
Status: Draft

---

# 1. Problem

A SaaS user signs up but does not know how to reach value.

The product may have:

- empty states;
- too many settings;
- no guided next step;
- no sample data;
- unclear workspace setup;
- no progress feedback;
- no connection between setup and user goal.

The result is low activation, poor retention and weak product validation.

---

# 2. Context

This Pattern applies to SaaS products with:

```text
signup
workspace creation
project creation
team setup
role selection
document upload
configuration steps
trial activation
first workflow completion
```

It is especially useful for B2B SaaS, vertical SaaS, collaboration tools, document platforms, AI tools and creator platforms.

---

# 3. Forces

Key forces:

```text
speed versus required setup
personalization versus friction
guidance versus user autonomy
product education versus workflow completion
data collection versus trust
activation metrics versus user experience
```

---

# 4. Recommended Model

Use an activation-centered onboarding model:

```text
User intent
Required context
Minimum setup
Guided first action
Activation moment
Progress feedback
Empty-state guidance
Support fallback
Measurement
```

Example:

```text
An independent producer signs up, creates a project, uploads pitch materials and receives a financing readiness checklist.
```

---

# 5. Onboarding Stages

Recommended stages:

```text
1. Signup
2. User intent capture
3. Workspace or account setup
4. First project or object creation
5. Guided first workflow
6. Activation moment
7. Next best action
8. Feedback and retention trigger
```

Each stage should remove unnecessary friction.

---

# 6. Activation Moment

Define the activation moment clearly.

Examples:

```text
first project created
first document uploaded
first report generated
first team member invited
first workflow completed
first readiness score received
first payment or pilot request submitted
```

The activation moment must connect to the product's core value proposition.

---

# 7. Empty States

Empty states should guide users toward action.

A strong empty state includes:

```text
what this area is for
why it matters
primary action
example or template
optional help
```

Bad empty state:

```text
No projects yet.
```

Better empty state:

```text
Create the first project to organize your financing materials and prepare a readiness review.
```

---

# 8. Measurement

Track onboarding metrics:

```text
signup_completed
workspace_created
first_project_created
first_document_uploaded
activation_completed
time_to_activation
onboarding_dropoff_step
invited_team_member
```

---

# 9. Final Principle

> Onboarding should be designed as a path to value, not as a tour of features.
