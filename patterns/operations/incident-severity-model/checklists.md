# Incident Severity Model — Checklists

Version: 0.1.0  
Status: Draft

---

# 1. Context Fit Checklist

```text
[ ] Production users exist
[ ] Critical workflows exist
[ ] Alerts exist
[ ] Customer support needs incident language
[ ] Escalation rules are needed
[ ] Postmortems are needed for major incidents
```

---

# 2. Severity Definition Checklist

```text
[ ] SEV1 is defined
[ ] SEV2 is defined
[ ] SEV3 is defined
[ ] SEV4 is defined
[ ] Security criteria are included
[ ] Data integrity criteria are included
[ ] Business impact criteria are included
```

---

# 3. Classification Checklist

```text
[ ] Customer impact assessed
[ ] Affected users or tenants estimated
[ ] Critical workflow impact assessed
[ ] Data risk assessed
[ ] Security risk assessed
[ ] Workaround availability checked
[ ] Duration and trend reviewed
```

---

# 4. Response Checklist

```text
[ ] Incident owner assigned
[ ] Technical lead assigned
[ ] Communication owner assigned if needed
[ ] Support notified if needed
[ ] Security lead added if needed
[ ] Update frequency defined
[ ] Escalation rule known
```

---

# 5. Communication Checklist

```text
[ ] Internal channel created if needed
[ ] Customer impact statement prepared
[ ] Known facts separated from assumptions
[ ] Next update time defined
[ ] Support script prepared if needed
[ ] Resolution communication planned
```

---

# 6. Postmortem Checklist

```text
[ ] Postmortem requirement checked
[ ] Timeline captured
[ ] Impact documented
[ ] Root cause analyzed
[ ] Detection gap reviewed
[ ] Corrective actions assigned
[ ] Follow-up tracked
```

---

# 7. Final Principle

> Incident severity checklists should make response decisions fast, factual and consistent.
