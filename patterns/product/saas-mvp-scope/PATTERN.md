# SaaS MVP Scope Pattern

Version: 0.1.0  
Status: Draft

---

# 1. Problem

A SaaS team needs to define a first product version without expanding into a full platform.

Without discipline, MVP scope grows through:

- nice-to-have features;
- multiple personas;
- speculative enterprise requests;
- overbuilt settings;
- automation where manual work would validate faster;
- dashboards before core usage exists;
- integrations before workflow value is proven.

The result is slower delivery and weaker learning.

---

# 2. Context

This Pattern applies when a team is creating:

```text
new SaaS MVP
private beta
paid pilot
first production version
feature-limited launch
vertical SaaS prototype
market validation product
```

---

# 3. Forces

Key forces:

```text
speed versus completeness
learning versus polish
manual service versus automation
single segment focus versus broad market
usable product versus perfect product
revenue validation versus feature depth
```

---

# 4. Recommended MVP Model

Use this model:

```text
Segment: one target customer group
Problem: one painful and frequent problem
Workflow: one end-to-end core workflow
Outcome: one measurable customer result
Activation: one moment where user receives value
Feature set: only what supports the workflow
Non-scope: explicitly excluded items
Learning: metric and feedback loop
```

---

# 5. MVP Inclusion Rule

A feature belongs in the MVP only if it directly supports at least one of these:

```text
core workflow completion
activation moment
payment or pilot validation
trust and security baseline
feedback collection
basic operational support
```

If it does not support one of these, it should be delayed.

---

# 6. MVP Exclusion Rule

Delay features that are:

```text
nice-to-have
speculative
multi-segment
enterprise-only before validation
advanced reporting
deep customization
complex automation
non-critical integrations
```

Document excluded items in a non-scope list so they do not disappear into ambiguity.

---

# 7. Manual Before Automated

For early MVPs, manual operations may be acceptable when:

- customer value can still be delivered;
- volume is low;
- manual work accelerates learning;
- automation cost is high;
- the process is not security-critical;
- manual handling is transparent and reliable.

Do not manually handle sensitive or compliance-critical work without controls.

---

# 8. Success Criteria

Define success before building.

Examples:

```text
10 qualified users complete onboarding
5 customers submit real projects
3 paid pilots start
60% of beta users reach activation
first workflow completed in under 10 minutes
at least 30% of discovery users ask for access
```

---

# 9. Final Principle

> MVP scope should protect learning speed from product ambition.
