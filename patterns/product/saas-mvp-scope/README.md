# SaaS MVP Scope Pattern

Version: 0.1.0  
Status: Draft  
Domain: Product  
Category: MVP Definition

---

# 1. Purpose

The SaaS MVP Scope Pattern defines a reusable model for deciding what belongs in a SaaS MVP and what must be delayed.

It helps teams focus on the smallest product version that can validate a real customer problem, deliver a clear outcome and produce market learning without overbuilding.

---

# 2. Problem

SaaS MVPs often become too large.

Common failures:

```text
too many personas
too many features
unclear activation moment
no measurable success criteria
unclear paid use case
overbuilt admin and settings
weak onboarding
no feedback loop
```

---

# 3. Recommended Solution

Define the MVP around one customer segment, one painful use case and one measurable success outcome.

The MVP should include:

```text
primary user
core problem
core workflow
activation moment
minimum feature set
manual fallback where acceptable
success metric
feedback loop
clear non-scope
```

---

# 4. Recommended When

Use this Pattern when:

- a SaaS idea is being shaped;
- an MVP is being planned;
- a product scope is becoming too broad;
- a roadmap needs prioritization;
- early customers or pilots must validate demand;
- engineering needs a focused delivery target;
- launch speed matters.

---

# 5. Avoid When

Avoid this Pattern when:

- the product is already mature;
- the objective is platform expansion;
- compliance requires a larger baseline;
- an enterprise contract already defines fixed scope;
- the work is a deep technical migration, not MVP discovery.
