# SaaS MVP Scope — Context

Version: 0.1.0  
Status: Draft

---

# 1. Best-Fit Context

This Pattern is best for early SaaS products that need customer validation before full-scale buildout.

Typical contexts:

```text
idea validation
MVP planning
founder-led beta
private pilot
first paid customer test
pre-seed product build
vertical SaaS prototype
```

---

# 2. Team Context

Recommended when:

- team is small;
- development capacity is limited;
- product direction is still uncertain;
- founders need fast learning;
- engineering needs clear priorities;
- sales or discovery feedback is still emerging.

---

# 3. Customer Context

Use this Pattern when the target market has:

- a clear painful problem;
- a reachable buyer or user;
- willingness to try an early product;
- a workflow that can be validated quickly;
- a valuable outcome that can be measured.

---

# 4. Technical Context

Useful when:

- architecture can start simple;
- integrations can be delayed;
- advanced automation is not required;
- a modular monolith is enough;
- manual back-office operations can support early learning.

---

# 5. Warning Signs

MVP scope is unhealthy when:

- more than one primary persona exists;
- success metric is unclear;
- launch depends on advanced settings;
- the roadmap is full of speculative requests;
- no one can describe the activation moment;
- every feature is labeled critical;
- the product cannot be shipped without months of extra work.

---

# 6. Final Principle

> MVP context should be narrow enough to produce clear evidence.
