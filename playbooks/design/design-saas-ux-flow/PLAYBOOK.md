# Design SaaS UX Flow — Playbook Definition

Version: 0.1.0  
Status: Premium Draft

---

# 1. Purpose

Guide a structured process for designing the user flow of a SaaS feature or workflow.

---

# 2. Trigger

A product requirement, feature idea or workflow needs to be converted into a screen-by-screen user experience.

---

# 3. Inputs

Useful inputs include:

- PRD;
- user stories;
- target personas;
- feature scope;
- business objective;
- user goal;
- roles and permissions;
- data requirements;
- technical constraints;
- brand or UI guidelines;
- existing product screens.

---

# 4. Outputs

Expected outputs include:

- UX flow summary;
- user journey;
- screen inventory;
- flow diagram description;
- screen state list;
- edge case list;
- permission behavior notes;
- UX risks;
- design handoff checklist.

---

# 5. Execution Summary

```text
1. Define user goal
2. Identify roles and permissions
3. Define entry points
4. Map the happy path
5. Identify required screens
6. Define screen states
7. Map edge cases and errors
8. Review data and system feedback
9. Validate usability and friction
10. Produce UX flow handoff
```

---

# 6. Completion Criteria

The Playbook is complete when:

- the primary user goal is clear;
- the happy path is mapped;
- required screens are identified;
- states and errors are documented;
- permission behavior is clear;
- edge cases are visible;
- the flow is ready for wireframe or prototype work.

---

# 7. Escalation or Fallback

Escalate when:

- product requirements are unclear;
- user roles conflict;
- workflow depends on unknown technical constraints;
- data model is unclear;
- accessibility requirements are missing;
- security-sensitive screens are involved;
- the flow creates excessive user friction.

---

# 8. Final Principle

> UX flow design should remove ambiguity before pixels, components or code are created.
