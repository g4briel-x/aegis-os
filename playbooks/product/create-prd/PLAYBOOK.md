# Create PRD — Playbook Definition

Version: 0.1.0  
Status: Premium Draft

---

# 1. Purpose

Guide a structured process for creating a Product Requirements Document.

---

# 2. Trigger

A product idea, feature, workflow or business requirement needs to be converted into implementation-ready requirements.

---

# 3. Inputs

Useful inputs include:

- product context;
- feature idea;
- business objective;
- target users;
- user problem;
- existing workflow;
- constraints;
- stakeholder request;
- technical notes;
- design notes;
- success metrics;
- known risks.

---

# 4. Outputs

Expected outputs include:

- complete PRD;
- problem statement;
- goals and non-goals;
- user stories;
- functional requirements;
- non-functional requirements;
- acceptance criteria;
- metrics;
- risks;
- dependencies;
- open questions.

---

# 5. Execution Summary

```text
1. Define context
2. Define problem
3. Define users
4. Define goals and non-goals
5. Define scope
6. Write user stories
7. Define functional requirements
8. Define non-functional requirements
9. Define acceptance criteria
10. Define metrics, risks and open questions
```

---

# 6. Completion Criteria

The Playbook is complete when:

- the product problem is clear;
- goals and non-goals are documented;
- users and use cases are identified;
- requirements are testable;
- acceptance criteria are specific;
- metrics are defined;
- risks and dependencies are visible;
- open questions are captured.

---

# 7. Escalation or Fallback

Escalate when:

- user problem is unclear;
- feature scope is unstable;
- technical feasibility is unknown;
- security or compliance impact exists;
- design flow is complex;
- stakeholder priorities conflict;
- acceptance criteria cannot be tested.

---

# 8. Final Principle

> A strong PRD reduces ambiguity before ambiguity becomes rework.
