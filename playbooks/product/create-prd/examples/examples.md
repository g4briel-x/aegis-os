# 1. Example — Client Portal Feature

## Trigger

A SaaS team wants to create a client portal where customers can view project status, invoices and documents.

## Expected Execution

The Playbook should guide the user to:

- define customer and admin personas;
- state the problem;
- define goals and non-goals;
- write user stories;
- define permissions;
- define requirements;
- include acceptance criteria;
- capture security and data risks.

## Expected Output

```text
Context
Problem
Goals
Non-goals
Users
User stories
Functional requirements
Acceptance criteria
Risks
Open questions
```

# 2. Example — Audiovisual Project Submission Feature

## Trigger

A platform needs a feature where creators submit audiovisual projects for financing review.

## Expected Execution

The Playbook should guide the user to:

- define creator and reviewer roles;
- define submission workflow;
- define required project fields;
- define file upload rules;
- define review states;
- define acceptance criteria;
- flag security and storage risks.

# 3. Example — Role-Based Access Feature

## Trigger

A SaaS product needs admin, manager and viewer roles.

## Expected Execution

The Playbook should guide the user to:

- define protected resources;
- write user stories per role;
- define permission matrix;
- create acceptance criteria;
- require security review.

# 4. Example — Dashboard Feature

## Trigger

A product team wants a dashboard showing project progress and key metrics.

## Expected Execution

The Playbook should guide the user to:

- define dashboard user;
- identify decisions the dashboard supports;
- define metrics;
- define empty states;
- define loading and error states;
- define success metrics.
