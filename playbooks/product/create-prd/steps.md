# Create PRD — Steps

Version: 0.1.0  
Status: Premium Draft

---

# 1. Step 1 — Define Product Context

Describe the product, market, current state and reason for the requirement.

Capture:

- product name;
- product stage;
- business context;
- existing workflow;
- stakeholder request;
- strategic relevance.

Output:

```text
Product context summary
```

---

# 2. Step 2 — Define the Problem

State the problem before defining the solution.

A strong problem statement includes:

- affected user;
- current situation;
- pain or limitation;
- consequence;
- reason this matters now.

Output:

```text
Problem statement
```

---

# 3. Step 3 — Identify Users and Personas

Define who the requirement serves.

Capture:

- primary user;
- secondary users;
- internal operators;
- admins;
- support teams;
- decision makers.

Output:

```text
User and persona summary
```

---

# 4. Step 4 — Define Goals and Non-Goals

Define what the feature must achieve and what it intentionally excludes.

Goals should be outcome-oriented.

Non-goals prevent scope creep.

Output:

```text
Goals and non-goals
```

---

# 5. Step 5 — Define Scope

Define the included product behavior.

Capture:

- included workflows;
- included screens or surfaces;
- included data;
- included roles;
- excluded features;
- release boundaries.

Output:

```text
Scope definition
```

---

# 6. Step 6 — Write User Stories

Write user stories for key user actions.

Recommended format:

```text
As a <user>, I want to <action>, so that <value>.
```

Each story should be small enough to design, build and test.

Output:

```text
User story list
```

---

# 7. Step 7 — Define Functional Requirements

Translate stories into specific product behavior.

Functional requirements should define:

- user actions;
- system behavior;
- validations;
- permissions;
- states;
- error handling;
- notifications;
- data changes.

Output:

```text
Functional requirements
```

---

# 8. Step 8 — Define Non-Functional Requirements

Define quality requirements.

Consider:

- performance;
- security;
- availability;
- accessibility;
- auditability;
- scalability;
- usability;
- compatibility.

Output:

```text
Non-functional requirements
```

---

# 9. Step 9 — Define Acceptance Criteria

Define how the team will know the requirement is satisfied.

Acceptance criteria should be:

- testable;
- specific;
- observable;
- connected to user value.

Output:

```text
Acceptance criteria
```

---

# 10. Step 10 — Define Metrics

Define success measures.

Possible metrics:

- activation rate;
- completion rate;
- conversion;
- retention;
- error reduction;
- time saved;
- support ticket reduction;
- revenue impact.

Output:

```text
Success metrics
```

---

# 11. Step 11 — Identify Risks, Dependencies and Open Questions

Capture uncertainty before implementation.

Document:

- product risks;
- technical risks;
- design risks;
- security risks;
- dependencies;
- unresolved decisions.

Output:

```text
Risks, dependencies and open questions
```

---

# 12. Step 12 — Assemble the PRD

Combine all validated sections into a structured PRD.

Output:

```text
Product Requirements Document
```

---

# 13. Final Principle

> PRD creation should move from context to problem, then from problem to testable requirements.
