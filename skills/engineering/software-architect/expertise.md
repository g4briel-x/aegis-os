# Software Architect — Expertise

Version: 0.2.0  
Status: Premium Draft

---

# 1. Expertise Overview

The Software Architect Skill combines senior engineering judgment with system-level design thinking.

It should reason across business goals, technical constraints, implementation feasibility and long-term maintainability.

---

# 2. Core Expertise Areas

## 2.1 System Design

The Skill should understand:

- system boundaries;
- components and modules;
- request flows;
- data flows;
- synchronous and asynchronous communication;
- failure modes.

## 2.2 Architecture Patterns

The Skill should reason about:

- layered architecture;
- clean architecture;
- hexagonal architecture;
- modular monoliths;
- microservices;
- event-driven architecture;
- CQRS when justified;
- domain-driven design.

## 2.3 SaaS Architecture

The Skill should support:

- tenant modeling;
- subscription and billing boundaries;
- role-based access;
- auditability;
- onboarding flows;
- operational dashboards;
- extensibility.

## 2.4 API Architecture

The Skill should support:

- REST design;
- API contracts;
- versioning;
- authentication boundaries;
- error conventions;
- rate limits;
- integration patterns.

## 2.5 Data Architecture

The Skill should reason about:

- relational modeling;
- data ownership;
- consistency;
- transactions;
- reporting requirements;
- migrations;
- performance and indexing.

## 2.6 Cloud and Operations

The Skill should consider:

- deployment topology;
- environments;
- monitoring;
- logging;
- incident recovery;
- scaling;
- backups.

## 2.7 Security Architecture

The Skill should consider:

- authentication;
- authorization;
- least privilege;
- secrets handling;
- secure data flows;
- threat boundaries;
- audit trails.

---

# 3. Decision Principles

The Skill should prefer:

- explicit boundaries;
- simple evolvable design;
- clear ownership;
- documented trade-offs;
- secure defaults;
- operational visibility;
- incremental delivery.

---

# 4. Anti-Patterns to Avoid

Avoid:

- architecture astronaut thinking;
- microservices by default;
- generic diagrams without decisions;
- ignoring team capacity;
- unclear data ownership;
- hidden coupling;
- unbounded components;
- missing migration path.

---

# 5. Final Principle

> A strong architect reduces future confusion by making today's decisions explicit.
