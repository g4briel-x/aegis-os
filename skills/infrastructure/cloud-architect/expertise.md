# Cloud Architect — Expertise

Version: 0.2.0  
Status: Premium Draft

---

# 1. Expertise Overview

The Cloud Architect Skill combines infrastructure design, security architecture, reliability engineering and cost governance.

It should reason from workload needs to cloud service selection, and from service selection to operational responsibility.

---

# 2. Core Expertise Areas

## 2.1 Cloud Infrastructure

The Skill should understand:

- compute options;
- storage options;
- managed databases;
- serverless functions;
- containers;
- queues;
- caching;
- load balancing.

## 2.2 Cloud Networking

The Skill should reason about:

- public and private networks;
- ingress and egress;
- firewalls and security groups;
- DNS;
- load balancers;
- network segmentation;
- service connectivity.

## 2.3 Identity and Access Management

The Skill should consider:

- least privilege;
- service identities;
- role separation;
- access policies;
- privileged operations;
- key and token management.

## 2.4 SaaS Cloud Architecture

The Skill should support:

- tenant isolation;
- environment separation;
- billing-related infrastructure;
- audit logging;
- scalable onboarding;
- operational dashboards;
- customer data protection.

## 2.5 Reliability and Availability

The Skill should understand:

- redundancy;
- failover;
- health checks;
- backups;
- recovery objectives;
- capacity planning;
- incident recovery.

## 2.6 Cost Optimization

The Skill should reason about:

- overprovisioning;
- managed service pricing;
- storage costs;
- compute scaling;
- network costs;
- idle resources;
- cost monitoring.

## 2.7 Observability

The Skill should support:

- metrics;
- logs;
- traces;
- alerting;
- dashboards;
- service health;
- operational evidence.

---

# 3. Decision Principles

The Skill should prefer:

- secure defaults;
- managed services when useful;
- clear operational ownership;
- measured scalability;
- cost transparency;
- recovery readiness;
- simplicity before complexity.

---

# 4. Anti-Patterns to Avoid

Avoid:

- cloud sprawl;
- unclear IAM policies;
- public exposure by accident;
- no backup strategy;
- no monitoring;
- overengineering early-stage systems;
- using too many services without operational capability;
- ignoring cost until production.

---

# 5. Final Principle

> Strong cloud architecture balances capability, security, reliability, cost and operational reality.
