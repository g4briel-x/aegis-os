# Incident Report Template — Checklists

Version: 0.1.0  
Status: Draft

---

# 1. Completeness Checklist

```text
[ ] Incident name is defined
[ ] Incident status is defined
[ ] Severity is assigned
[ ] Owner is assigned
[ ] Current status is written
[ ] Customer impact is documented
[ ] Affected systems are listed
[ ] Timeline is started
[ ] Actions taken are recorded
[ ] Next steps are assigned
```

---

# 2. Impact Checklist

```text
[ ] Affected users or tenants are estimated
[ ] Affected workflows are listed
[ ] Business impact is reviewed
[ ] Security or privacy impact is reviewed
[ ] Data integrity impact is reviewed
[ ] Workaround is documented if available
```

---

# 3. Communication Checklist

```text
[ ] Incident channel is defined
[ ] Support is notified if customer-facing
[ ] Customer communication decision is recorded
[ ] Status update cadence is defined
[ ] Next update time is visible
[ ] Final resolution communication is planned
```

---

# 4. Closure Checklist

```text
[ ] Resolution time is documented
[ ] Validation is completed
[ ] Customer impact end time is documented
[ ] Postmortem requirement is checked
[ ] Evidence links are attached
[ ] Incident commander approves closure
```

---

# 5. Final Principle

> Incident report checklists should keep facts, impact and ownership visible.
